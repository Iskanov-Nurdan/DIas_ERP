# DIAS — бизнес‑логика по разделам (UI / RBAC)

Этот документ описывает **сценарии и правила**, которые реализованы на бэкенде (Django/DRF), без догадок.
Ключ доступа раздела берётся из `user.accesses[]` (см. `GET /api/me`).

---

## 1) Пользователи и роли (`users`)

### Пользователи
- **Системные пользователи** защищены: не отдаются/не редактируются через API.
- `name` должен быть уникальным; имя `admin` зарезервировано для системного пользователя.
- При создании пользователь получает `email` вида `<name_slug>@local.dias`.

### Доступы пользователя (вкладки)
- Эндпоинт `PATCH /api/users/{id}/access/` делает **полную замену** списка ключей.
- Ключи валидируются по списку `ACCESS_KEYS` в настройках.

### Роли
- Системные роли нельзя удалить/изменить.

---

## 2) Линии (`lines`) и смены (`my_shift`, `shifts`)

### Смена на линии (операционная)
- На линии может быть **только одна открытая** смена.
- Смена может быть **в паузе**; пока пауза активна:
  - нельзя создавать выпуск партии (`POST /api/batches/`)
  - нельзя создавать/редактировать замесы на линии (`POST/PATCH /api/production/recipe-runs/`)
- История событий смены хранится в `LineHistory` (open / params_update / pause / resume / close).

### Открытие/закрытие
- `POST /api/lines/{id}/open/` требует размеры `height,width,angle_deg`.
- `POST /api/lines/{id}/close/`:
  - если размеры переданы — используются они
  - если размеры не переданы — берутся последние параметры смены из истории; если параметров нет — ошибка.

### Личная смена (без линии)
- `POST /api/shifts/open/` без `line_id` открывает личную смену (`line=null`).
- `GET /api/shifts/my/` показывает только личную открытую смену.
- Заметки `POST /api/shifts/notes/` возможны **только** при открытой личной смене.

### Жалобы
- Полная лента доступна суперпользователю или с ключом `shifts`.
- Иначе пользователь видит только свои жалобы и те, где он упомянут.

## 3) Материалы (`materials`)

### Справочник сырья
- `unit` канонизируется в `kg` или `g`. Любое другое значение запрещено.
- Смена `unit` через `PATCH` может быть запрещена, если сырьё уже используется в приходах/движениях/рецептах.
- Удаление запрещено, если сырьё «в использовании» (приходы, движения, рецепты, химия, производство).

### Приход (партии сырья)
- Количество в `POST /api/incoming/` задаётся **в единицах карточки сырья** (`kg` или `g`),
  но в партии хранится в кг (для FIFO).
- FIFO и движения формируются на сервере.

### Остатки и движения
- `GET /api/materials/balances/` считает остатки через FIFO (`material_stock_kg`).
- `GET /api/materials/movements/` агрегирует:
  - incoming (партии прихода)
  - writeoff (списания из `MaterialStockDeduction`) с типами причин (production/chemistry/other)

## 4) Химия (`chemistry`)

### Справочник хим. элементов
- У элемента есть **состав** (нормы сырья на единицу выпуска) — `recipe_lines`.
- Изменение состава делается отдельным `PATCH` **только** с полем `recipe_lines` (без смешивания с другими полями).
- Удаление запрещено при наличии:
  - партий выпуска химии
  - ссылок из рецептов
  - факта расхода в замесах/производстве

### Выпуск химии
- `POST /api/chemistry/elements/produce/`:
  - списывает сырьё FIFO по составу
  - создаёт `ChemistryBatch` (партия химии)
  - при недостатке остатков возвращает конфликт (`409`) по доменным кодам.

### Задания
- `POST /api/chemistry/tasks/` создаёт задание на выпуск.
- `POST /api/chemistry/tasks/{id}/confirm/` выполняет выпуск и переводит задание в `done`.
- Выполненное (`done`) задание удалить нельзя.

## 5) Профили и рецепты (`recipes`)

### Профили
- Удаление запрещено, если есть связанные рецепты или партии производства.
- Вместо удаления предполагается `is_active=false`.

### Рецепты
- Рецепт хранит нормы **на 1 метр** и не списывает склад сам по себе.
- При `POST/PATCH` компоненты передаются массивом; при обновлении — старые компоненты удаляются и создаются заново.
- Удаление запрещено при наличии партий/заказов/замесов; вместо удаления — деактивация.

### Availability
- `GET /api/recipes/{id}/availability/` рассчитывает потребность и сравнивает с остатками по FIFO
  отдельно для сырья и химии.

## 6) Производство (`production`) и ОТК (`otk`)

### Партия производства (выпуск)
- Создание `POST /api/batches/`:
  - требует открытой и активной смены на линии
  - требует открытой смены пользователя на этой линии
  - проверяет, что рецепт относится к профилю и имеет компоненты
  - списывает сырьё/химию FIFO и рассчитывает себестоимость на сервере

### Отправка в ОТК
- `POST /api/batches/{id}/submit-for-otk/` переводит партию в очередь ОТК только из корректного статуса.

### Приёмка ОТК
- `POST /api/batches/{id}/otk_accept/`:
  - проверяет, что партия ещё `pending`
  - `accepted + defect` должны равняться `pieces`
  - если есть брак — причина обязательна
  - создаёт записи ОТК и переводит партию в завершённое состояние
  - создаёт складские строки ГП по результатам ОТК

### Замесы (RecipeRun)
- Замес — план ёмкостей и факта расхода для интерфейса.
- При создании замеса автоматически создаётся/обновляется связанная pending‑партия, и списание идёт по той же общей логике, что и у `POST /api/batches/`.
- Удаление замеса делает reverse списаний и удаляет pending‑партию, но только пока партия ещё не прошла ОТК.

## 7) Склад ГП (`warehouse`)

### Источник склада
- Строки склада создаются из результатов ОТК производственной партии.

### Резерв
- `POST /api/warehouse/batches/reserve/` переводит строку в `reserved`, но только если резервируется **весь остаток строки**.

### Упаковка
- `POST /api/warehouse/batches/package/`:
  - работает только для строк в форме учёта `unpacked` и статусе `available`
  - создаёт новую строку склада в форме `packed`
  - качество/причина брака переносятся из исходной строки; «смешивание партий» запрещено.

---

## 8) Клиенты (`clients`) и продажи (`sales`)

### Клиенты
- Удаление запрещено, если есть связанные продажи (сохранение истории).

### Продажи
- Продажа может быть:
  - без привязки к складу (просто финансовая запись)
  - со складом (тогда выполняется операция по складу по количеству)
- Если продажа уже привязана к складу, нельзя:
  - менять `warehouse_batch`
  - менять `quantity`/`quantity_input`
  - менять `stock_form`/`piece_pick`
  (чтобы не ломать складскую историю; вместо этого — новая продажа или отмена)

Документы:
- `GET /api/sales/{id}/nakladnaya/` (HTML черновик)

## 9) Аналитика (`analytics`)

Все эндпоинты — `GET` и работают по единому набору фильтров периода/разрезов.
Подробно: `FRONTEND_ANALYTICS_API.md` (в корне репозитория).

## 10) Журнал действий (`activity`)

Система аудита пишет события по сущностям (create/update/delete) с `request_id`, привязкой к сменам и payload изменений.
